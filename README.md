# Games Member Club: SQL & Database Integrity Showcase

A full-stack interactive dashboard application showcasing database-level business logic, relational schemas, automated ledger triggers, check constraints, and analytical views using **MySQL** and **Streamlit**.

---

## 🎮 Project Overview

This project demonstrates how core business rules and database constraints can be enforced directly inside the database engine. Enforcing logic at the schema layer ensures data integrity, consistency, and auditability, independent of any external frontend or API services.

The application simulates a gaming club setup where:
1. Members can check in/out of gaming stations (VR, PC, PS5, Xbox, Board Games).
2. The ledger tracks all deposits, payments, and refunds.
3. Database triggers automatically handle loyalty tier upgrades, guard against insufficient funds, calculate session costs, and generate invoice ledger transactions.

---

## 🛠️ Advanced SQL Features Showcased

### 1. Automated Loyalty Tier Progression (`after_transaction_insert`)
Every time a deposit transaction is logged, a trigger automatically updates the member's wallet balance, calculates their lifetime deposits, and upgrades their membership tier accordingly:
*   **Bronze**: Lifetime deposits < $50
*   **Silver**: Lifetime deposits $\ge$ $50
*   **Gold**: Lifetime deposits $\ge$ $200
*   **VIP**: Lifetime deposits $\ge$ $500

### 2. Database-Level Insufficient Funds Guard (`before_booking_insert`)
Before a member can start an ongoing booking, the database checks if their wallet balance is sufficient to cover at least **one hour** of the selected station's rate. If insufficient, it throws a SQL `SIGNAL SQLSTATE '45000'` error and rejects the insert.

### 3. Automatic Cost Calculation & Invoice Generation (`before_booking_update` & `after_booking_complete`)
*   **Cost Calculation**: When a booking is updated to `Completed`, the engine automatically calculates the duration (down to the second) and sets the `total_cost` based on the station's rate.
*   **Ledger Generation**: Once the booking is completed, a post-update trigger automatically writes a negative `Payment` transaction record to the ledger, which cascades to deduct the member's wallet.

## 📊 Analytical Views Implemented

*   **`view_active_sessions`**: Real-time snapshot of active players, their stations, and session start times.
*   **`view_member_financials`**: Financial ledger view showing lifetime deposits, lifetime spent, and current wallet balance for each member.
*   **`view_game_performance`**: Analytical view aggregating usage count, total hours played, and revenue generated by each gaming station.

## 📂 Repository Structure

*   `schema.sql`: Contains the complete MySQL schema definition (tables, constraints, triggers, views).
*   `seed.sql`: Rich mock data representing realistic club activity (members, games, deposits, completed & ongoing bookings).
*   `app.py`: Beautiful Streamlit web application to visually run trigger scenarios, analyze views, and run custom queries.
*   `db_setup.sh`: Script to spin up a standalone, passwordless MySQL instance on port `33066` and apply the schema and seed data.
*   `db_stop.sh`: Script to stop the local MySQL server.
*   `.gitignore`: Properly configured to ignore MySQL binary database folders and Python cache.

## ⚡ Setup & Run Instructions

### Prerequisites
*   macOS (or standard Linux environment)
*   Python 3.x
*   MySQL installed on the host system (the setup script uses the system `mysqld` command to launch a local server)

### 1. Clone & Install Dependencies
```bash
pip install pymysql streamlit pandas
```

### 2. Initialize & Start the Database
Run the setup script to initialize the custom data directories, launch MySQL on port `33066`, and apply the schema/seed data:
```bash
./db_setup.sh
```

### 3. Launch the Dashboard
```bash
streamlit run app.py
```
This will open the dashboard in your default browser (usually at `http://localhost:8501`).

### 4. Stop the Database Server
When finished, stop the database instance:
```bash
./db_stop.sh
```
